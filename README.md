# Pipelines Without Pipelines

> How we built a data-pipeline engine with **no pipeline object** — a finite state
> machine you never wrote, implemented by the scripts themselves.

📍 *Presented at **Commit** — Paris, 18 June 2026.*

This repo holds the talk, the blog post, and a **runnable demo pipeline** behind
Windmill's asset-graph data pipelines: the pipeline graph isn't declared anywhere,
it *emerges* from what each script reads and writes, and the same WASM-compiled
parser that draws the graph in the browser also drives dispatch in the backend.

## The demo pipeline (`f/orders`)

`demo-pipeline/` is a **Windmill CLI sync repo**: a polyglot, asset-driven pipeline
(Bun + DuckDB + Python) where membership, lineage edges, triggers, and cascade
subscriptions are all derived from the script bodies at deploy time — there is no
separate pipeline-definition file. See **`demo-pipeline/README.md`** for the DAG, the
annotation reference, and a step-by-step demo script.

Deploy it to your own workspace with the
[Windmill CLI](https://www.windmill.dev/docs/advanced/cli):

```bash
npm install -g windmill_cli                          # or see the docs for other installers
wmill workspace add <name> <workspace-id> <base-url> # authenticate to your workspace

cd demo-pipeline
wmill sync push --auto-metadata                      # deploys f/orders/* (regenerates locks/schemas)
```

Then open the pipeline view (folder **f/orders**) and run a source script to watch
the cascade. Edit a script locally and `wmill sync push` again — the graph updates
from the code, with nothing else to maintain.

> **Requirements:** a Windmill instance on a build that includes the data-pipelines /
> asset-graph feature. The pipeline references workspace **data tables** and **S3
> object storage** — configure those in your workspace (the scripts create the tables
> they need on first run).

## What's the idea, in one paragraph

A classic Windmill **flow** is an explicit finite state machine: a `FlowValue` DAG
walked by a generic interpreter, state persisted to Postgres after every step. A
**data pipeline** is the photographic negative — there's no pipeline object at all.
Each script declares what assets it reads and writes; the graph is the union of those
reads and writes, reconstructed on demand by a `SELECT`. Execution is dispatch, not
orchestration: when a script writes an asset, every subscriber fires. The keystone is
a single asset parser compiled twice — native Rust in the backend (authoritative
dispatch at deploy) and WebAssembly in the editor (the live graph as you type) — so
the script definitions are at once the program and the picture. Zero drift, by
construction.
